# Postback + Status Groups

Brokers report lead status changes by POSTing to GambChamp's postback endpoint. The status mapping layer translates each broker's raw status string into a canonical status that the platform understands universally.

## Postback intake

Brokers send status updates to:

```
POST /api/v1/postbacks/:brokerId
```

Requests are authenticated via HMAC signature. The broker signs the request body using the shared `Broker.postbackSecret`, and the server verifies the signature before processing. Unsigned or incorrectly signed requests are rejected with `401`.

Each postback carries a `status` field — a free-form string the broker defines (for example: `"ftd"`, `"Qualified"`, `"duplicate_lead"`, `"hold"`). These raw strings vary widely between brokers and cannot be compared across integrations without normalization.

## Raw → canonical mapping

The `StatusMapping` table maps each `(brokerId, rawStatus)` pair to a `CanonicalStatus` row. When a postback arrives:

1. `classifyLeadStatus(brokerId, rawStatus)` looks up the mapping.
2. If a mapping exists, the canonical status name is returned.
3. If no mapping exists, the lead's `canonicalStatus` is written as `"unmapped"`.

The canonical status then drives:

- `Lead.canonicalStatus` — stored on the lead record.
- `Lead.state` transition — e.g. a `CONVERTED`-category canonical status transitions the lead to `FTD`.
- `LeadEvent` emission — `STATUS_UPDATED` with the canonical and raw values.
- `LeadDailyRoll.canonicalStatus` — used by the analytics rollup for cross-broker reporting.

## The four categories

Every canonical status belongs to one of four categories:

| Category | Meaning |
|----------|---------|
| `NEW` | Broker accepted the lead; not yet converted |
| `QUALIFIED` | Lead has registered and verified identity |
| `REJECTED` | Broker declined the lead |
| `CONVERTED` | First-time deposit confirmed |

There are 20 seeded `CanonicalStatus` rows (5 per category) covering the most common broker status semantics. Custom canonical statuses can be added via the admin page.

## Coverage metric

The **coverage metric** shows the percentage of observed raw statuses (from the last 30 days of postbacks) that have a mapping for a given broker. A broker with 95% coverage means 5% of postbacks are landing as `"unmapped"` and are not being correctly classified.

Unmapped statuses are visible on the broker's Status Mapping page alongside their observed frequency. You should map any status that appears more than once in 30 days.
