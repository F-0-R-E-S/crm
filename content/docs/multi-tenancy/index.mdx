# Multi-Tenancy

GambChamp is a multi-tenant platform. Each tenant is a fully isolated white-label instance with its own affiliates, brokers, leads, flows, and branding. Tenants cannot see or affect each other's data.

## Tenant identity

Every tenant has a unique **slug** (e.g. `acme`). All tenant-scoped Prisma models carry a `tenantId` field that maps to the `Tenant` table. The application enforces isolation at the database middleware layer â€” see [Concepts: withTenant](/docs/multi-tenancy/concepts#withtenant).

## 3-domain hostname pattern

GambChamp uses a three-part hostname structure to route requests to the correct tenant and expose the correct surface:

```
<role>.<slug>.<ROOT_DOMAIN>
```

| Role prefix | Hostname example | Surface |
|-------------|-----------------|---------|
| `network` | `network.acme.example.com` | Dashboard, super-admin, docs, tRPC, auth |
| `autologin` | `autologin.acme.example.com` | Autologin deep-link pages + `/api/v1/autologin/*` |
| `api` | `api.acme.example.com` | REST intake only â€” `/api/v1/*` |

`src/middleware.ts` parses the `x-forwarded-host` header, resolves the tenant slug to a `tenantId` via `resolveTenantFromHost`, and propagates the tenant context via an `x-tenant-slug` header throughout the request.

**Path mismatches** return 404. For example, a request to `api.acme.example.com/dashboard` is rejected because the `api` role does not expose the dashboard surface.

### Disabling the 3-domain pattern

When `ROOT_DOMAIN` is empty or unset, every hostname maps to `tenant_default` with the `network` role. This is the default for local development and single-tenant self-hosted deployments.

## Tenant branding

Each tenant can configure its brand identity via `Tenant.theme`:

| Field | Purpose |
|-------|---------|
| `brandName` | Display name in the dashboard header and emails |
| `logoUrl` | URL of the tenant's logo image |
| `primaryColor` | Main brand colour (hex) |
| `accentColor` | Secondary / highlight colour (hex) |
| `legalLinks` | Footer links for privacy policy, terms, etc. |

The `TenantBrandingStyle` RSC injects `--brand` and `--accent` CSS variables into the dashboard layout. Tenants edit their branding at **Dashboard â†’ Settings â†’ Branding** (shortcut `V`).

## Super-admin cross-tenant access

The `SUPER_ADMIN` role can perform CRUD operations across all tenants. Super-admin pages live at `/super-admin/tenants` and use the `superAdminProcedure` tRPC procedure, which skips `withTenant` scoping. Regular admin users are confined to their own tenant.
